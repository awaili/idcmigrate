External Agent Executor — Contract & Instructions

What an executor must implement to work with idc-migrate. Full source contract: docs/agent-executor.md. Reference implementation: idc/executor_mock/app.py.

In one sentence: an executor is an external HTTP service that idc-migrate outsources code-scan / assessment / modify work to. It receives trigger requests, does the work async, and pushes results back to idc-migrate via callbacks. Results feed reconsolidate, wave planning, and match confidence.

one configured endpoint connect via the Manage executor panel (Code & DB tab) or IDC_EXECUTOR_URL / IDC_EXECUTOR_TOKEN env.

Auth & conventions

  • Both directions use Authorization: Bearer <token>. The shared secret is IDC_EXECUTOR_TOKEN (idc-migrate validates pushes with it; the executor validates incoming triggers with the same value).
  • Content-Type application/json; charset=utf-8. Timestamps ISO-8601 UTC (...Z).
  • Request direction base = IDC_EXECUTOR_URL (e.g. http://executor.internal:8090). Push direction base = the idc-migrate web URL (e.g. https://mig.zaymuc.com).
  • All triggers are async: respond 202 {job_id, status:"pending"} immediately, then call back when done.

The 5 actions (idc-migrate → executor)

These are the endpoints the executor must serve. All bearer-authed, all async (202 + job_id).

EndpointActionProduces (via callback)
POST /v1/scanscan a repo (read-only) → findingspushes ChangeJob (done); scan itself does not push a profile
POST /v1/combsynthesize a CodeProfile from findingsChangeJob done + PUT /api/code-profiles/{app_id}
POST /v1/modifyapply a concrete changes[] list (idc-migrate builds it)ChangeJob done with patch_ref
POST /v1/db-scan F5scan a DB host's schema/SQL (Oracle/SQLServer → MySQL)ChangeJob done + PUT /api/db-profiles/{db_server_id}
POST /v1/runtime-containerize F9infer a Dockerfile from runtime inventory (no source repo)ChangeJob done + PUT /api/code-profiles/{app_id} with source=runtime-derived
GET /v1/jobs/{job_id}poll a job's statusreturns the ChangeJob
GET /healthliveness + identity{ok, service, version, ...} — powers the status badge / idc doctor
modify is the only action with side effects. Default mode=plan (dry-run, produce a patch_ref only); mode=execute actually writes to the repo (commit/PR).

POST /v1/scan

{"app_id": "orders-svc", "repo_url": "git@gitlab:trade/orders-svc.git",
 "branch": "master", "callback": "https://mig.zaymuc.com/api/code-profiles/orders-svc"}
→ 202 {"job_id": "cjob-...", "status": "pending"}

POST /v1/modify

The changes array is built by idc-migrate from the app's CodeProfile + operator overrides — it tells the executor exactly which file, which line, which literal (old) → which value (new). Do not re-scan; do not guess.

{"app_id": "orders-svc", "repo_url": "git@gitlab:trade/orders-svc.git",
 "branch": "cloud-ready", "mode": "plan",
 "scope": ["db_connection", "service_discovery"],
 "changes": [
   {"category": "db_connection", "kind": "endpoint",
    "file": "src/main/resources/application.yml", "line": 12,
    "title": "Parameterize DB host",
    "evidence": "url: jdbc:mysql://10.0.4.20:3306/orders",
    "old": "10.0.4.20:3306", "new": "${ORDERS_DB_HOST}:3306",
    "effort": "low", "description": "Replace hardcoded 10.0.4.20 with ${DB_HOST}"},
   {"category": "secrets_in_repo", "kind": "secret",
    "file": "src/main/resources/application.yml", "line": 13,
    "title": "Move DB password to secret ref",
    "evidence": "password: hunter2",
    "old": "hunter2", "new": "${ORDERS_DB_PASSWORD}",
    "effort": "low", "description": "Inline password → secret ref"}
 ],
 "notes": []}
→ 202 {"job_id": "cjob-...", "status": "pending"}
If a change.old or change.new is empty, idc-migrate couldn't extract the literal. The executor must locate it by file+line itself, or skip it and report ChangeJob.status=error. Never invent a replacement value.

POST /v1/db-scan (F5)

{"db_server_id": "db-oracle-01", "source_engine": "oracle",
 "target_engine": "tdsql", "callback": "https://mig.zaymuc.com/api/db-profiles/db-oracle-01"}
→ 202 {"job_id": "...", "status": "pending"}

db_server_id is the DB host's stable identity (hostname, lowercase) — not Server.id, which changes every rebuild.

Push callbacks (executor → idc-migrate)

The executor calls these as it works. All bearer-authed with IDC_EXECUTOR_TOKEN.

EndpointWhen
POST /api/change-jobsheartbeat + terminal state — pending→running→done/error. Upsert by id. Use summary for live progress text.
PUT /api/code-profiles/{app_id}when comb/scan/runtime-containerize produces a CodeProfile (upsert by app_id)
PUT /api/db-profiles/{db_server_id}when db-scan produces a DBConversionProfile (upsert by hostname)
POST /api/apps/{app_id}/questionsraise a question to the operator when a value can't be resolved (see Questions)
Callback order on completion: POST /api/change-jobs (status=done) first, then the profile PUT (only for comb/scan/db-scan that produce a profile).

CodeProfile — the assessment (must produce for comb)

Pushed via PUT /api/code-profiles/{app_id}. Data model: idc/core/models.py::CodeProfile.

{
  "app_id": "orders-svc",
  "repo_url": "git@gitlab:trade/orders-svc.git",
  "branch": "master",
  "scan_id": "cjob-1a2b3c4d5e",
  "scanner": "idc-executor/0.3.0",
  "scanned_at": "2026-07-12T08:30:00Z",
  "language": "java", "runtime": "jdk8", "framework": "spring-boot-2.1",
  "cloud_readiness": 0.42,
  "migration_pattern": "replatform",
  "refactor_effort": "medium",
  "code_deps": ["user-svc", "inventory-svc"],
  "network_endpoints": ["10.0.4.20:3306", "redis01.dc1:6379"],
  "required_changes": [
    {"title": "Parameterize DB host", "category": "db_connection",
     "file": "src/main/resources/application.yml", "effort": "low",
     "description": "Replace hardcoded 10.0.4.20 with ${DB_HOST}"}
  ],
  "blockers": ["Quartz scheduler assumes host crontab (scheduled_job, blocker)"],
  "findings": [
    {"category": "db_connection", "severity": "high",
     "file": "src/main/resources/application.yml", "line": 12,
     "message": "hardcoded DB host",
     "evidence": "url: jdbc:mysql://10.0.4.20:3306/orders",
     "remediation": "use ${DB_HOST} from env"}
  ],
  "summary": "Spring Boot 2.1 on JDK8; rehost blocked by Quartz-on-crontab; needs DB host param + service-discovery swap."
}

Required fields

  • cloud_readiness ∈ [0,1] — from findings' severity/count. A blocker finding must push this below 0.5.
  • migration_pattern ∈ rehost | replatform | refactor | rewrite.
  • refactor_effort ∈ low | medium | high.
  • code_deps — app ids actually called in code (HTTP/RPC/message/shared lib). The key signal that back-fills the dependency DAG.
  • blockers — hard blockers (human text, goes into match rationale).
  • required_changes[] — each {title, category, file, effort, description}.
  • findings[] — each {category, severity, file, line, message, evidence, remediation}.
  • scanned_at / scanner / scan_id.

Finding categories (use these)

hardcoded_ipservice_discoverydb_connectionstateful_localscheduled_jobsecrets_in_repobaremetal_assumptionnetwork_dependencyos_dependencylegacy_runtimeconfig_coupling

F5 DB categories: plsql_compatdb_feature_gapdb_size_complexity. Severity: low | medium | high | blocker.

DBConversionProfile (F5 — for db-scan)

Pushed via PUT /api/db-profiles/{db_server_id}. Keyed by hostname (stable), not Server.id.

{
  "db_server_id": "db-oracle-01",
  "source_engine": "oracle", "target_engine": "tdsql",
  "difficulty": "C",            // A | B | C  (Ora2Pg A–C / DMS SC grade)
  "est_man_days": 40.0,
  "review_objects": ["PKG_ORDERS", "PKG_BILLING", "TRG_AUDIT_LOG"],
  "blockers": ["PL/SQL packages with no MySQL equivalent (DBMS_LOCK, UTL_FILE)"],
  "reverse_replication": true,  // keep a reverse channel until cutover finalizes
  "auto_convert_pct": 0.62,     // 0..1 — share the rule engine can auto-convert
  "scan_id": "cjob-db1", "scanned_at": "2026-07-12T08:00:00Z",
  "summary": "Oracle → TDSQL: heavy PL/SQL, hard conversion, reverse replication recommended."
}

idc-migrate use: C → strong confidence dip + force replatform; B → small dip; A → no dip, just annotated.

ChangeJob — the audit/heartbeat record

{
  "id": "cjob-1a2b3c4d5e", "app_id": "orders-svc", "kind": "scan",
  "repo_url": "git@gitlab:trade/orders-svc.git", "branch": "master",
  "status": "done",             // pending | running | done | error
  "patch_ref": "",              // commit/PR/artifact url (modify)
  "summary": "scanned 1248 files, 12 findings",
  "error": "",
  "created_at": "2026-07-12T08:29:00Z", "finished_at": "2026-07-12T08:30:00Z"
}

Push status=running repeatedly during long work, updating summary for live progress; terminal done/error goes through the same endpoint.

Context the executor can pull (read-only GET, no bearer)

While working, the executor may fetch more context from idc-migrate:

  • GET /api/workloads/{app_id} — the app's tier, depends_on, server_ids (blast radius).
  • GET /api/apps/{app_id}/targets — matched Tencent targets for the app's servers {product, spec, region, confidence} (e.g. DB matched to CDB → parameterize JDBC to CDB-style host, not bare IP).
  • GET /api/apps/{app_id}/resolve?kind=ip&old=10.0.5.99 — "what should this old value become?" → {new, source, confidence}. source ∈ override | match-derived | default | unknown. If source=unknown, the executor MUST ask the operator (next section) — never guess.
  • GET /api/code-profiles/{app_id} — the app's current profile.

Questions — don't guess, ask

When a value can't be resolved (resolve returns unknown, a change.old/new is empty, or a new hardcoded value is found that's not in the profile), the executor raises a question instead of guessing or silently stalling.

POST /api/apps/{app_id}/questions   (bearer)
{"job_id": "cjob-...", "kind": "choice",        // value | choice | confirm
 "prompt": "What should '10.0.4.20:3306' in src/.../application.yml:12 (db_connection) be replaced with?",
 "options": ["${DB_HOST}:3306"],
 "context": {"file": "src/main/resources/application.yml", "line": 12,
             "category": "db_connection", "old": "10.0.4.20:3306", "new": ""}}
→ {"id": "qst-...", "status": "pending"}

Then poll GET /api/questions/{id} until status=answered, read answer, continue. (Operator answers via the UI: POST /api/questions/{id}/answer; /skip to dismiss.)

Quality rules

  • Traceable — every finding carries file+line+evidence; no vague claims.
  • Deterministic first — rule checks must be reproducible; use MigraQ only for semantic judgment, and always pin its conclusion to an evidence-backed finding.
  • Idempotent — same (app_id, repo_url, branch) → upsert by app_id, no duplicates.
  • Truncate — evidence ≤ 512 chars; ≤ 500 findings/app (keep top by severity).
  • Fail loudly — on error push ChangeJob.status=error + message; never swallow.
  • Default read-only — modify is dry-run unless the caller passes mode=execute.
  • Don't guess values — ask a Question instead.

Lifecycle & errors

  • Triggers are async: 202 + job_id; results via callback or GET /v1/jobs/{id}.
  • Timeout: IDC_EXECUTOR_TIMEOUT (default 600s) bounds the HTTP connection, not the job — long work is expressed via ChangeJob.status.
  • Concurrency: a new scan for the same app_id overwrites the old profile; same job_id re-push is upsert.
  • Status codes: 400 schema, 401 auth, 404 unknown app, 409 already-running (optional), 422 repo unreachable, 5xx internal.

Minimal viable executor

The smallest useful executor implements:

  1. GET /health → {ok:true, service, version}.
  2. POST /v1/scan + POST /v1/comb → return 202 + job_id, then in the background: POST /api/change-jobs (running → done) and PUT /api/code-profiles/{app_id} with a real CodeProfile.
  3. POST /v1/modify → apply the given changes[] (literal replace by file+line), produce a patch_ref; mode=plan dry-run only.
  4. GET /v1/jobs/{id} → return the job record.

db-scan (F5) and runtime-containerize (F9) are optional until you need them.

Reference implementation: idc/executor_mock/app.py — ~370 lines, implements the whole contract with in-memory jobs and real callbacks. Run it with idc executor-mock (port 8090), then point idc-migrate at it via the Manage executor panel (http://127.0.0.1:8090 + token) and Save & test. A production executor swaps the mock's canned answers for a real scanner (AST/regex rules + optional MigraQ for semantic calls) and a real repo writer (commit/PR for modify).
