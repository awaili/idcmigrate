External Agent Executor — Contract & Instructions

What an executor must implement to work with idc-migrate. Full source contract: docs/agent-executor.md. Reference implementation: idc/executor_mock/app.py.

In one sentence: an executor is an external worker that idc-migrate outsources code-scan / assessment / modify work to. It exposes nothing to the network — it pulls work from idc-migrate's POST /api/executor/tasks/claim, does the work async, and pushes results back to idc-migrate over /api/*. Results feed reconsolidate, wave planning, and match confidence.

pull mode the executor only needs outbound to idc-migrate (it can sit behind NAT/firewall). Self-register with POST /api/executors/register (lands pending, no token); an operator approves you in the Manage-executors panel and idc-migrate mints + shows your token once for them to relay back to you. Then poll /api/executor/tasks/claim with that bearer. Set IDC_PUBLIC_URL on idc-migrate so it can bake the feedback callback into each task.

Quick start — the 5-call happy path (read this first)

The exact call sequence a from-scratch executor implements. Every call is executor → idc-migrate over HTTPS with Authorization: Bearer $IDC_EXECUTOR_TOKEN. Reference loop: idc/executor_mock/app.py.

  1. Configure — set IDC_EXECUTOR_TOKEN (shared secret; or the ex-… token minted on approval) + IDC_CALLBACK_BASE (idc-migrate's public HTTPS base, e.g. https://mig.zaymuc.com) + repo access (SSH key / git creds, for scan/comb/modify). The executor brings its own LLM (§docs/agent-executor.md §2.7).
  2. (Named executor only) Register — POST /api/executors/register {id} → pending; operator approves → you get an ex-… token out-of-band. The default executor skips this (token pre-shared).
  3. Poll for work — POST /api/executor/tasks/claim {} → {task_id, kind, payload, lease_until} or {task_id:null, status:"idle"}. The claim holds a 15-min lease; kind dispatches the worker (see task kinds + the per-kind payloads below).
  4. Heartbeat (long jobs) — POST /api/executor/tasks/{id}/lease {} before lease_until, or the task is requeued and another executor may reclaim it.
  5. Do the work + push results — run the per-kind logic, then POST /api/change-jobs (running → done) then the kind's artifact PUT (e.g. PUT /api/code-profiles/{app_id} with a CodeProfile). Use payload.callback when non-empty, else {IDC_CALLBACK_BASE}/api/….
  6. Mark terminal — POST /api/executor/tasks/{id}/complete {"status":"done","summary":"…","result_ref":""} (or "status":"error","error":"…" — never leave a claimed task stranded). Then loop to 3.
That is the whole loop: claim → (lease) → push change-jobs + artifact → complete. Everything below is the precise reference for each call's body/response/errors; the task-kind payloads tell you exactly which fields each kind carries and which PUT it pushes back.

Auth & conventions

  • The executor exposes nothing to the network — it only makes outbound calls to idc-migrate. Every call uses Authorization: Bearer <token>; the shared secret is IDC_EXECUTOR_TOKEN. The token also identifies the executor (resolved to an executor id) and gates which tasks it may claim.
  • Content-Type application/json; charset=utf-8. Timestamps ISO-8601 UTC (...Z).
  • Base = idc-migrate's public HTTPS URL (e.g. https://mig.zaymuc.com) — the executor pulls /api/executor/tasks/claim from it and pushes results back to it. The executor must be able to reach it from its network; idc-migrate never calls back.
  • Tasks are async: a trigger returns 200 {task_id, status:"pending"} immediately; the executor pulls + completes it.
  • Auth is required on every /api/* path — the live idc-migrate runs behind a web login gate, so the executor must send Authorization: Bearer $IDC_EXECUTOR_TOKEN on every claim, complete, push, and context pull (including the read-only GETs below). A call without a bearer returns 401; the bearer is the only option a headless executor has (no browser session).
  • No /health probe. idc-migrate cannot reach the executor, so there is no liveness probe — the executor proves it is alive by polling /api/executor/tasks/claim. executor_status reports pull mode (reachable=null); real liveness is "an executor is claiming tasks".
  • Multiple executors are supported, and they self-register. An executor enrolls with POST /api/executors/register public (no bearer yet) — optionally gated by X-Enroll-Secret when IDC_ENROLL_SECRET is set. It lands pending with no token; an operator approves it (POST /api/executors/{id}/approve) and idc-migrate mints its token and returns it once for the operator to relay out-of-band (the executor never holds a secret it invented). The default executor is the single operator-configured one (IDC_EXECUTOR_TOKEN / the panel). A trigger's executor_id picks a target: empty = the pool (any approved executor may claim it); "default" or a named id = target that executor specifically (404 if unknown, 409 if still pending). POST /api/executors/{id}/rotate-token re-issues an approved executor's token. Each executor authenticates with its own token — idc-migrate accepts a bearer matching any approved executor's token; a pending/unapproved token gets 401.

Pulling work (executor → idc-migrate)

The executor polls POST /api/executor/tasks/claim (bearer-authed); idc-migrate atomically hands it one eligible task with the full payload (the work + the feedback callback). The executor runs it, pushes results back to idc-migrate's public HTTPS URL, then POST /api/executor/tasks/{id}/complete. The callback in the payload is authoritative; when empty the executor builds {IDC_CALLBACK_BASE} + path itself — see Push callbacks.

Pulling a task — POST /api/executor/tasks/claim

POST /api/executor/tasks/claim   (Authorization: Bearer $IDC_EXECUTOR_TOKEN)
{}
→ 200 {"task_id": "tsk-...", "kind": "scan", "status": "claimed",
        "payload": {"app_id": "orders-svc", "repo_url": "git@gitlab:trade/orders-svc.git",
                    "branch": "master",
                    "callback": "https://mig.zaymuc.com/api/code-profiles/orders-svc"},
        "executor_id": null, "lease_until": "2026-07-15T10:15:00Z"}
   (or → 200 {"task_id": null, "status": "idle"} when the queue is empty)

One task per call — loop with a short sleep when idle. The payload.callback is where results go back. When done: POST /api/executor/tasks/{task_id}/complete {"status":"done","summary":"..."} — the status string is exactly "done" or "error" (anything else like "completed"/"success" → 422, task left stranded). Renew the lease with /lease on long jobs.

kind=modify — the changes list

The changes array is built by idc-migrate from the app's CodeProfile + operator overrides — it tells the executor exactly which file, which line, which literal (old) → which value (new). Do not re-scan; do not guess.

Push callbacks (executor → idc-migrate)

The executor calls these as it works, against idc-migrate's public HTTPS URL. All bearer-authed with IDC_EXECUTOR_TOKEN. The target URL comes from the task payload's callback field when idc-migrate baked one in (authoritative — push directly to it), else the executor builds {IDC_CALLBACK_BASE} + path itself.

Method is fixed per endpoint — do not improvise. Every profile/artifact/result push below is a PUT; only /api/change-jobs and /api/apps/{app_id}/questions are POST. The callback URL fixes the path, not the method — a POST to a PUT endpoint returns 405 Method Not Allowed and your result is lost. Likewise POST /api/executor/tasks/{id}/complete takes status = exactly "done" or "error" — not completed/success/finished (those get 422 and strand the task).

PUT /api/db-profiles/{db_server_id}when db-scan produces a DBConversionProfile (upsert by hostname)

Callback order on completion: POST /api/change-jobs (status=done) first, then the profile/artifact PUT (only for actions that produce one).
Reachability: idc-migrate's push endpoints sit behind its public nginx HTTPS ingress (e.g. https://mig.zaymuc.com/api/*) and accept the bearer token before the login gate — no extra port or certificate needed. idc-migrate sends its own public URL as callback on every trigger (set IDC_PUBLIC_URL on the idc-migrate side, or the Manage-executor panel); when it's unset the executor must configure IDC_CALLBACK_BASE to that same public domain. Either way the executor pushes over the public internet to idc-migrate's HTTPS address.

Required fields

  • cloud_readiness ∈ [0,1] — from findings' severity/count. A blocker finding must push this below 0.5.
  • migration_pattern ∈ rehost | replatform | refactor | rewrite.
  • refactor_effort ∈ low | medium | high.
  • code_deps — app ids actually called in code (HTTP/RPC/message/shared lib). The key signal that back-fills the dependency DAG.
  • blockers — hard blockers (human text, goes into match rationale).
  • required_changes[] — each {title, category, file, effort, description}.
  • findings[] — each {category, severity, file, line, message, evidence, remediation}.
  • scanned_at / scanner / scan_id.

Optional agent (codex) grounded fields — path A optional

Only present when the executor's optional codex pass is on (EXECUTOR_CODEX_SCAN=true, off by default). After the rule engine scans, the executor runs codex exec -s read-only over the cloned checkout to find code-level blockers / patterns / cross-file deps the regexes miss — grounded in the actual source. The pass never writes the repo, never blocks the rule-only push (any failure → rule-only + skip), and counts against EXECUTOR_TASK_TIMEOUT (bump to ~900 on big repos). Requires codex CLI + ollama installed where the executor runs. See executor/agent_scan.py.

  • agent_findings[] — same shape as findings[]; category is one of the rule categories above OR agent_insight (a semantic/cross-file issue no rule category fits). Grounded in real file:line; capped at 20, evidence ≤ 512.
  • agent_blockers[] — hard blockers the agent found (one line each). idc-migrate's audit_match / seven_r_strategy / review_plan weight these HIGHER than the regex blockers (read from the actual repo, not a regex).
  • agent_summary — 2-4 sentence grounded assessment.

Context the executor can pull (read-only GET, bearer-authed)

While working, the executor may fetch more context from idc-migrate. All of these are bearer-authed with IDC_EXECUTOR_TOKEN (same secret as the pushes) — under the live login gate there is no anonymous access, so a GET without a bearer returns 401:

  • GET /api/workloads/{app_id} — the app's tier, depends_on, server_ids (blast radius).
  • GET /api/apps/{app_id}/targets — matched Tencent targets for the app's servers {product, spec, region, confidence} (e.g. DB matched to CDB → parameterize JDBC to CDB-style host, not bare IP).
  • GET /api/apps/{app_id}/resolve?kind=ip&old=10.0.5.99 — "what should this old value become?" → {new, source, confidence}. source ∈ override | match-derived | default | unknown. If source=unknown, the executor MUST ask the operator (next section) — never guess.
  • GET /api/code-profiles/{app_id} — the app's current profile.

Questions — don't guess, ask

When a value can't be resolved (resolve returns unknown, a change.old/new is empty, or a new hardcoded value is found that's not in the profile), the executor raises a question instead of guessing or silently stalling.

POST /api/apps/{app_id}/questions   (bearer)
{"job_id": "cjob-...", "kind": "choice",        // value | choice | confirm
 "prompt": "What should '10.0.4.20:3306' in src/.../application.yml:12 (db_connection) be replaced with?",
 "options": ["${DB_HOST}:3306"],
 "context": {"file": "src/main/resources/application.yml", "line": 12,
             "category": "db_connection", "old": "10.0.4.20:3306", "new": ""}}
→ {"id": "qst-...", "status": "pending"}

Then poll GET /api/questions/{id} until status=answered, read answer, continue. (Operator answers via the UI: POST /api/questions/{id}/answer; /skip to dismiss.)

Quality rules

  • Traceable — every finding carries file+line+evidence; no vague claims.
  • Deterministic first — rule checks must be reproducible; use MigraQ only for semantic judgment, and always pin its conclusion to an evidence-backed finding.
  • Idempotent — same (app_id, repo_url, branch) → upsert by app_id, no duplicates.
  • Truncate — evidence ≤ 512 chars; ≤ 500 findings/app (keep top by severity).
  • Fail loudly — on error push ChangeJob.status=error + message; never swallow.
  • Default read-only — modify is dry-run unless the caller passes mode=execute.
  • Don't guess values — ask a Question instead.

Lifecycle & errors

  • Triggers are async: a trigger returns 200 {task_id, status:"pending"}; the task waits in the queue until an executor claims it.
  • Lifecycle: pending → claimed → done|error. A claim holds a 15-min lease — renew via /lease on long jobs, or the task is requeued and another executor may reclaim it. idc-migrate also mirrors the terminal state into change-jobs.
  • Concurrency: a new scan for the same app_id overwrites the old profile; multiple executors may poll the same queue — the atomic claim hands each task to exactly one.
  • Status codes: 400 schema, 401 auth/unknown token, 404 unknown task/executor, 409 not-owned (complete/lease on a task the caller doesn't own), 422 repo unreachable (reported via task error).

Minimal viable executor

The smallest useful executor is a poll loop (it exposes no HTTP server):

  1. Loop: POST /api/executor/tasks/claim (bearer). On task_id:null sleep briefly and retry.
  2. For kind=scan|comb: clone the repo, scan it, then push back POST /api/change-jobs (running → done) and PUT /api/code-profiles/{app_id} (to the callback in the payload) with a real CodeProfile.
  3. For kind=modify: apply the payload's changes[] (literal replace by file+line), produce a patch_ref; mode=plan dry-run only.
  4. Always finish with POST /api/executor/tasks/{task_id}/complete {"status":"done"|"error",...} (renew via /lease if the work is long). On any worker error, complete with status=error — never leave a task claimed.

db-scan (F5), runtime-containerize (F9), and discover-repos are optional until you need them (discover-repos only matters if the operator uses the Code tab's "Discover repos in a git group" — it enumerates repos inside a group/org url via the SCM API with the executor's own creds).

Reference implementation: idc/executor_mock/app.py — a pull poller that claims tasks, runs the canned scan/comb/modify/… logic, pushes results back, and completes. Run it with idc executor-mock (set IDC_MOCK_CALLBACK to idc-migrate's base + IDC_EXECUTOR_TOKEN to the shared secret, both registered in the Manage executors panel). A production executor swaps the mock's canned answers for a real scanner (AST/regex rules + optional MigraQ for semantic calls) and a real repo writer (commit/PR for modify).
